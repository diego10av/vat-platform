## 2 · Working well (no toco)

1. **Pipeline mechanics** — 8 stages (`cold_identified → cold_warm →
   first_touch → meeting_held → proposal_sent → in_negotiation → won
   / lost`). Kanban view, `weighted_value_eur` STORED GENERATED,
   `stage_entered_at` for velocity metrics. Merge cold→main pipeline
   (stint 64.Q.7) eliminates context-switching.
2. **Contact model sophistication** — lifecycle ≠ role_tags
   (Salesforce/Veeva split), `engagement_level` auto-computed from
   `last_activity_at`, employment history (`started_at` / `ended_at`
   on the junction), consent tracking (Art. 7).
3. **Billing** — partial payments via `crm_billing_payments` journal,
   `outstanding` GENERATED, explicit VAT (not back-calculated),
   `line_items` JSONB, approval workflow, credit notes, PDF generation.
4. **Actionable widgets** — every widget passes Rule §11
   (clicking → act):
   - `ForecastWidget` (weighted Q close + drill-through filter)
   - `WipWidget` (top 5 unbilled matters)
   - `ActionsDueWidget` (ranked NBA feed with inline mark-done + snooze)
   - `KeyAccountHealthWidget` (risk pill per KA)
5. **Inline editing** — uniform pattern ported across companies,
   contacts, opportunities, matters, tasks (single-field PATCH routes,
   no page reload, error reverts).

## 3 · Bugs / broken (concretos, fichero:línea)

| Fichero | Issue |
|---|---|
| `src/app/api/crm/contacts/[id]/route.ts` (no endpoint) | No bulk-update for `lifecycle_stage`. Migrar 30 leads → prospects requires one-at-a-time edits. The single-row PUT accepts the field; the bulk drawer simply doesn't expose it. |
| `src/app/crm/contacts/[id]/page.tsx` | Column `engagement_override` exists in DB; UI doesn't expose it. Can't manually mark a "dormant" contact as "active" without DB edit. |
| `src/app/crm/outreach/page.tsx:18` | Dead redirect post-stint 64.Q.7 (when outreach merged into main pipeline). Bookmark-courtesy at best; should be a `notFound()` or removed. |
| `src/app/api/crm/opportunities/bulk-update/route.ts:13` | `bulk-update` ALLOWED_FIELDS deliberately excludes `company_id` + `primary_contact_id` per "Stint 63.H — heavy action". Defensible, but `lifecycle_stage` for contacts has no parallel and ought to. |

(The `won_reason` / `loss_reason` form gap mentioned in the prior
exploration was **fixed in this commit cycle** as part of stint 91.)

## 5 · Clutter para retar (Rule §11)

Cosas que muy probablemente no aportan valor real al dogfood diario.
**Recomiendo discutirlas en la próxima sesión** — la duda real es
si Diego las usa nunca.

1. **`FirstTimeBanner` en `/crm`** — onboarding nudge dismissible.
   Diego ya pasó por él. Borrar.
2. **Calendar cross-module (`/crm/calendar`)** — UNE eventos tax-ops
   en una vista del CRM. Viola Rule §14 (independencia modular).
   Discutir si tiene uso real o es ruido.
3. **`UpcomingThisWeekWidget`** — read-only, sin botón de acción.
   Vanity por Rule §11 (cambiar → no actúo).
4. **`lead_score` + `lead_score_reasoning`** — columnas pobladas por
   un batch mensual de Haiku; la UI NO los muestra. **Estamos pagando
   API calls por algo invisible.** Decidir: exponer en UI o quitar
   el batch.
5. **`/crm/help` (235 líneas)** — leído una vez al setup; nunca más.
   Mover a `docs/CRM_GUIDE.md` y dejar un link en sidebar.
6. **Help page sub-sections sobre features removidas** (chat,
   email drafter, portal) si sobreviven después de la purga del
   reset 2026-05-05.

## 6 · Top-5 cambios de impacto (priorizados)

Si Diego quiere atacar el siguiente sprint, esto es lo que daría
más por menos:

| # | Cambio | Scope | Por qué |
|---|---|---|---|
| **1** | **Win/loss reporting widget** | medium (6-8 h) | Counts YTD, win rate %, avg deal size por source. Una sola SQL query. Alto signal sobre la salud del negocio. Ahora que `won_reason` se captura (mig 091), tiene materia prima. |
| **2** | **Pipeline velocity widget** | medium (4-6 h) | "Avg N días en proposal_sent" usando `stage_entered_at`. Identifica deals estancados + forecast realista. |
| **3** | **Bulk lifecycle_stage update** | small (2-3 h) | Bulk drawer: añadir dropdown lifecycle_stage. API ya lo acepta. Desbloquea "mark 30 old leads as lost" en un click. |
| **4** | **`engagement_override` UI en contact detail** | small (1-2 h) | Cierra el caso "matter de larga duración sin actividades, pero el contacto está activo". Un checkbox. |
| **5** | **Probabilidad por stage** | small (2-3 h) | Default `probability_pct` cuando se cambia stage (stage `proposal_sent` → 70%, etc.). Reduce fricción en forecasting. |

## 8 · Lo que NO recomiendo construir

- **Email-mention auto-parse / phone auto-extract** — coste alto
  (requiere LLM por nota), valor incierto para single-user dogfood.
- **RFP / proposal generation** — out of scope hasta que haya un
  cliente real esperando una propuesta automatizada.
- **Multi-user roles** — explícitamente parqueado en CLAUDE.md §8.
