# Accessibility — audit + fix backlog

> Snapshot of accessibility concerns with what's fixed and what's
> queued. Target: pass axe-core DevTools scan at AA level without
> critical/serious violations.
>
> Last audit: **2026-04-18**

---

## ✅ Fixed in this pass

### Skip-to-content link

Invisible-until-focused `<a href="#main-content">` at the top of
`AppShell`. First Tab press from page load reveals the link; Enter
jumps past the 232px sidebar straight to the page content.
Matches WCAG 2.4.1 Bypass Blocks.

### Dialog labelling

Every modal that has `role="dialog"` + `aria-modal="true"` also has
`aria-labelledby` pointing to the heading:
- `BulkEditModal` → `#bulk-edit-title`
- `ExcelImportModal` → `#excel-import-title`
- `AttachmentsModal` → `#attachments-title`
- `FeedbackModal` → `#fb-modal-title`

### Icon-button labels

`IconBtn` atom used across the declaration page now emits `aria-label`
from its `title` prop so the existing call sites get coverage for
free. Individual close buttons in modals also gained `aria-label`.

### Decorative SVGs

Inline SVGs inside buttons labelled with `aria-label` now carry
`aria-hidden="true"` so assistive tech doesn't announce them twice.

### SearchBar

- Trigger button `aria-label="Open global search (⌘K)"`.
- Dropdown `role="dialog" aria-label="Search"`.
- Input `aria-label="Find entity, declaration, or provider"`.
- Decorative magnifying-glass SVG `aria-hidden="true"`.

### DocRow keyboard access

`DocRow` was a clickable `<div>` with no keyboard equivalent — screen
readers couldn't trigger it, Tab skipped over it.
Now: `role="button"`, `tabIndex={0}`, `aria-pressed` reflecting
selection state, `aria-label` composed from filename + status, and
an `onKeyDown` that fires `onSelect` on Enter/Space. Preserves the
interior Retry button because we can't nest buttons.

Added `focus-visible:ring-2` so keyboard focus is actually visible
on the row.

## 🚧 Known-but-deferred

### 1. Focus trap in modals

None of our modals trap Tab. A keyboard user inside the "Share for
approval" modal can Tab out to elements behind the backdrop.

**Fix recipe:**
- Inside the modal's `useEffect`, on open: query all focusable
  children, focus the first.
- On Tab at the last focusable: cycle back to first.
- On Shift+Tab at the first: cycle back to last.
- Can extract to a reusable `useFocusTrap(ref)` hook.

**Risk:** low. Library-level work, not app behaviour. Leaving for
a focused a11y pass.

### 2. Focus return on modal close

When a modal closes, focus should return to the element that opened
it (Esc in the "Share" modal should return to the "Share for approval"
button, not to document.body).

**Fix recipe:**
- Each modal stores a ref to the opener, restores focus on unmount.
- Same `useFocusTrap` hook can expose this.

### 3. Toast accessibility

`ToastProvider` renders a fixed-position rack of toasts. Needs
`role="status"` + `aria-live="polite"` for info/success, and
`role="alert"` + `aria-live="assertive"` for errors.

**Fix recipe:** update `ToastRack` + `ToastItem` with the right
ARIA roles based on `kind`.

### 4. Table-row keyboard access

The declaration-detail review table, the entities list, the
declarations list: clicking a row opens detail. No keyboard
equivalent. Mouse-only.

**Fix recipe:** same treatment as DocRow — either `<tr tabIndex={0}
onKeyDown={Enter/Space}>` or wrap the row content in a `<Link>` with
spans so the whole row is a link.

### 5. Color contrast audit

`text-ink-faint` and `text-ink-muted` classes on near-white
backgrounds may fall below WCAG AA 4.5:1 for small text. Need a
Lighthouse / axe run to confirm specific instances, then bump the
design tokens up one step where it fails. The three-level ink
palette (ink / ink-soft / ink-muted / ink-faint) should have
AA-passing contrast on the lightest two at minimum.

### 6. Form field label associations

Most of our form inputs use the pattern `<label><span>Label</span>
<input/></label>` (wrapping) which is fine for screen readers. A
handful use only a visually-styled heading + sibling input — those
need `htmlFor` + `id` or `aria-labelledby`. Need to grep for `<input`
and spot-check.

### 7. Focus-visible styles missing on some buttons

Several custom-styled buttons have `focus:outline-none` without a
`focus-visible:ring-*` replacement. Means keyboard users can't tell
what's focused. Global fix: a Tailwind layer class that adds the
ring token, then apply everywhere.

### 8. Screen-reader live regions for async state

When the chat streams a reply, or when an extraction finishes, the
UI changes silently for a screen reader. `aria-live="polite"`
regions at the points of change would announce "New message",
"Extraction complete".

## Testing

Manual for now — recommended flow before declaring a release a11y-clean:

1. Open each main page in Chrome.
2. Run **Lighthouse → Accessibility** (command-palette or DevTools panel).
3. Target score **≥ 95**. Anything below is a regression.
4. Run **axe DevTools** extension, scan each page.
5. Zero critical/serious issues is the bar.

Automated: deferred. Adding a11y checks to CI needs either Playwright
with axe (requires a running stack) or static analysis tools like
`eslint-plugin-jsx-a11y`. The latter is cheap to add; queued.
